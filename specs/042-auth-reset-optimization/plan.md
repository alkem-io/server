# Implementation Plan: Authorization Reset Performance Optimization

**Branch**: `042-auth-reset-optimization` | **Date**: 2026-03-19 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/042-auth-reset-optimization/spec.md`

## Summary

Optimize the full-platform authorization reset from ~4,400 individual RabbitMQ messages with per-entity load-compute-save cycles to batched messages (~90 at batch size 50), split the deep-nested profile query to avoid cartesian product explosion, add two missing FK indexes, consolidate redundant `saveAll()` cascades into single top-level persistence, and introduce bulk UPDATE for authorization policy rows.

## Technical Context

**Language/Version**: TypeScript 5.3, Node.js 22 LTS (Volta 22.21.1)
**Primary Dependencies**: NestJS 10, TypeORM 0.3, Apollo Server 4, RabbitMQ (via `@nestjs/microservices`)
**Storage**: PostgreSQL 17.5 — authorization policies stored with JSONB columns (`credentialRules`, `privilegeRules`)
**Testing**: Vitest 4.x — unit tests in `src/`, integration/e2e in `test/functional/`
**Target Platform**: Linux server (Docker / Kubernetes)
**Project Type**: Single NestJS monolith
**Performance Goals**: ≥60% wall-clock reduction on full auth reset; ≥95% reduction in RabbitMQ messages; ≥50% fewer DB round-trips per user reset
**Constraints**: Identical authorization policy state post-reset (backward compatibility); peak memory must not increase; existing retry semantics preserved
**Scale/Scope**: ~4,400 entities across users/accounts/organizations in a typical deployment; ~3k TypeScript source files

## Phase 0: Research Findings

All unknowns resolved — see [research.md](./research.md) for full details.

**Key decisions**:
1. **Batch payload**: Extend `AuthResetEventPayload` with `ids: string[]` array alongside existing `id` for backward compatibility during rolling deployment
2. **Split query strategy**: Replace single deep LEFT JOIN with 3 sequential queries: (a) profile + authorization, (b) references + tagsets + visuals with auth, (c) storageBucket + documents + document.tagsets with auth
3. **Bulk UPDATE**: Use TypeORM QueryBuilder `update().set().whereInIds()` for homogeneous policy batches, falling back to chunked `save()` for heterogeneous updates
4. **Index migration**: Standard `CREATE INDEX` / `DROP INDEX` migration pattern per existing conventions
