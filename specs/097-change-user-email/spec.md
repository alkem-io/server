# Feature Specification: Platform Admin Change User Login Email (No Verification)

**Feature Branch**: `097-change-user-email`
**Created**: 2026-05-13
**Last Updated**: 2026-05-18 (smaller MVP — email-ownership verification moved to spec 098)
**Status**: Draft
**Input**: User description: "implement https://github.com/orgs/alkem-io/projects/50/views/8?pane=issue&itemId=187119757&issue=alkem-io%7Cserver%7C6064"

**Related Issues**:

- alkem-io/server#6064 — Allow changing a user's login email while preserving their Alkemio profile

**Related Specs**:

- **098 — Self-Service Email Change** (companion spec): adds the user-initiated path **with email-ownership verification** (`meUserEmailChangeBegin` + `userEmailChangeConfirm`, the `email_change_pending` entity, the token lifecycle, and the multi-step `initiated → confirmed → committed` state machine). 098 ships as a follow-up; this spec is the MVP that ships first.

**Why this spec ships first, and why it skips email-ownership verification**: The motivating incident in #6064 is that platform admins were performing direct database edits to change user emails when users contacted support after losing access to their old mailbox. In that support flow, **identity verification of the user is performed manually by the admin out-of-band** (support call, prior account context, ID check, or other process the admin already follows). The admin's platform-admin authorization is therefore the only gate this spec enforces; the platform itself does NOT send a confirmation message to the new mailbox and does NOT require the new mailbox to prove ownership before the change is committed. This trade-off is acceptable here because (a) the admin already takes responsibility for the manual verification per current support practice, (b) the immediate ask of the issue is to stop the direct database edits — not to replace the support process with platform-mediated verification, and (c) the post-commit security-signal notification at the **old** address still gives the original user the opportunity to detect and escalate an erroneous or malicious change. The self-service flow in spec 098 introduces the full platform-mediated verification path for the case where no admin is in the loop and ownership of the new mailbox MUST be proven by the platform.

## User Scenarios & Testing _(mandatory)_

### User Story 1 — Platform Admin Changes a User's Login Email On Their Behalf (Priority: P1)

As a platform administrator, I want to change the login email of another user (e.g., when the user has lost access to their old mailbox and contacts support) so that I can restore their access without resorting to direct database edits, while remaining accountable in the audit trail.

**Why this priority**: This is the only user-facing capability in this spec. It is the support flow that replaces the ad-hoc database edits that motivated this feature. The admin is expected to have verified the user's identity out-of-band before invoking this capability.

**Independent Test**: A platform admin invokes the email-change mutation for another user's account with a new address. The platform validates the new address (format + uniqueness), commits the change on both the Alkemio user record and the identity provider in a single synchronous operation, invalidates the user's existing identity-provider sessions, sends a security-signal notification to the **old** address, and records every step in the audit trail. The change is observable end-to-end without any direct database edits. The user can subsequently log in only with the new address; the old address no longer authenticates.

**Acceptance Scenarios**:

1. **Given** a platform admin acting on behalf of another user, **When** the admin invokes the email-change mutation with a new address, **Then** the platform validates the new address (format + uniqueness) and, if validation passes, commits the change on both the Alkemio user record and the identity provider in a single synchronous operation, recording the action in the audit trail.
2. **Given** the change has committed, **When** the subject user attempts to log in with the new address, **Then** authentication succeeds; **When** they attempt to log in with the old address, **Then** authentication is rejected.
3. **Given** the change has committed, **When** the subject views their profile, **Then** all memberships, contributions, and historical references are intact and continue to reference the same user identity.
4. **Given** a non-admin user, **When** they attempt to invoke the email-change mutation for someone else, **Then** the platform rejects the request with an authorization error and no change occurs.
5. **Given** the change has committed, **When** the audit log is queried for the subject, **Then** the admin's identity, the subject's identity, the old address, the new address, the timestamp, and a `committed` outcome are recorded.

---

### User Story 2 — Atomic Two-Side Commit With Rollback on Partial Failure (Priority: P1)

As a platform operator, I want the email change to either fully succeed across both Alkemio and the identity provider or fully fail with no partial state, so that the two systems cannot silently drift out of sync the way they did under the legacy database-edit approach.

**Why this priority**: Drift between the Alkemio user record and the identity provider is the root failure mode the issue exists to fix. Even a perfect admin-authorization flow is unsafe if a mid-commit failure leaves Alkemio updated but the identity provider unchanged (or vice versa). Without this guarantee, the feature reintroduces the very problem it was created to solve.

**Independent Test**: Simulate a failure on the second-side write (e.g., the identity provider is unreachable or returns an error) after the first side has been updated, and verify that the first-side change is rolled back so the user's login email is unchanged on both sides. A clear error is surfaced to the admin. If the rollback itself also fails, the system records a drift-detected audit entry and surfaces a distinct error so the admin can reconcile via a dedicated mutation.

**Acceptance Scenarios**:

1. **Given** the admin has invoked the email-change mutation, **When** the platform updates the identity provider successfully but the Alkemio write fails, **Then** the platform reverses the identity-provider update so both sides reflect the old email, surfaces a clear error to the admin, and records a `rolled_back` audit entry.
2. **Given** the admin has invoked the email-change mutation, **When** the identity-provider write fails after the retry budget is exhausted, **Then** no change is made on either side (commit ordering is Kratos-first; the Alkemio write never starts), and the platform surfaces a clear error and records a `rolled_back` audit entry.
3. **Given** any partial-failure recovery has run, **When** the subject user next attempts to log in, **Then** the old email continues to work (the system is in its pre-change state), and the admin can retry the mutation.
4. **Given** both the forward commit AND the compensating rollback exhaust their retry budgets, **When** the system enters drift, **Then** an audit entry with outcome `drift_detected` is recorded capturing the observed value on each side, a Winston error log entry and an Elastic APM error capture are emitted with marker `email_change_drift_detected`, and a distinct error is surfaced to the admin identifying that drift remains and that the dedicated reconciliation mutation MUST be used.

---

### User Story 3 — Conflict and Validation Errors (Priority: P2)

As an admin initiator of an email change, I want to be told immediately when the proposed new address is invalid or already in use, so that no change is attempted on a target value that could never be assigned.

**Why this priority**: P2 because the system would still be correct if uniqueness were only enforced when the side-writes attempt to take effect, but a clean pre-write validation gives the admin immediate feedback and avoids triggering the FR-009 compensating rollback for a benign uniqueness conflict. Also avoids accidentally writing a duplicate-email row that would fail the database unique constraint with a less-friendly error surface.

**Independent Test**: Submit an admin-initiated email change with (a) a malformed address, (b) the subject's current address, and (c) an address already belonging to another Alkemio user or identity provider identity. Each MUST be rejected before any side-write is attempted; no audit entry of a successful commit MUST be produced.

**Acceptance Scenarios**:

1. **Given** a malformed new email address, **When** the change is requested, **Then** the platform rejects with a validation error.
2. **Given** a new email identical to the subject's current login email, **When** the change is requested, **Then** the platform rejects with a "no change" error.
3. **Given** a new email already in use by another Alkemio user or identity provider identity, **When** the change is requested, **Then** the platform rejects with a "conflict" error, and the error MUST NOT confirm or deny which user owns the conflicting address (to avoid enumeration).

---

### User Story 4 — Audit Trail of Email Changes (Priority: P3)

As a platform operator (and, for the user's own benefit, as the user themselves), I want every email change recorded with who initiated it, who the subject was, the old address, the new address, and when it happened, so that we can investigate incidents and answer "did anyone touch this account?" later.

**Why this priority**: Audit logging is critical for security investigations and for support, but the system is functional without it; therefore P3. It MUST land with the feature, not retrofitted.

**Independent Test**: After any email change attempt — successful commit, rolled-back failure, drift-detected double-failure, drift-resolved reconciliation, or rejected validation/conflict — an audit entry is retrievable from a paginated GraphQL query under the `platformAdmin` surface, keyed by the subject user. The entry includes initiator, subject, old address, new address, timestamp, outcome, and (where applicable) a non-leaky failure reason. The query never exposes any PII beyond `{ id, displayName }` for the initiator and subject.

**Acceptance Scenarios**:

1. **Given** a successfully committed email change, **When** the audit log is queried, **Then** an entry exists with initiator, subject, old address, new address, timestamp, and outcome = `committed`.
2. **Given** a partial-failure rollback, **When** the audit log is queried, **Then** an entry exists with outcome = `rolled_back` and a non-leaky failure reason describing the side(s) on which the failure occurred.
3. **Given** a drift-detected double-failure, **When** the audit log is queried, **Then** an entry exists with outcome = `drift_detected` capturing the observed values on each side and a non-leaky failure reason.
4. **Given** an admin-driven drift reconciliation, **When** the audit log is queried, **Then** an entry exists with outcome = `drift_resolved` capturing the canonical email selected.
5. **Given** a rejected attempt (validation, conflict), **When** the audit log is queried, **Then** an entry exists with the corresponding `rejected_validation` or `rejected_conflict` outcome and a non-leaky failure reason (e.g., the conflict entry MUST NOT reveal that another user owns the conflicting address).

---

### Edge Cases

- The subject user's old mailbox is already unreachable when the change is initiated. The admin-on-behalf path explicitly supports this case — the admin's authorization, not the subject's mailbox access, is what gates the operation.
- The new email is identical (case-insensitive) to the current email. The platform MUST treat addresses case-insensitively for equality and conflict checks.
- The identity provider returns a non-error but unexpected response shape (e.g., the identity id no longer exists). The platform MUST treat this as a failure of the commit and trigger rollback rather than recording success.
- The admin invokes the mutation twice in quick succession for the same subject and the same new address. Both invocations validate and attempt to commit; if the first succeeds, the second sees the second-time `no-change` (the new email is now the current) and rejects with a `no-change` error.
- The subject user's Alkemio account is deactivated or deleted between admin invocation and the side-writes completing. The platform MUST detect the absent subject (or the missing identity-provider identity) on the read step and reject with a `not-found` error before any side-write.
- The post-commit security-signal notification to the old address fails to send. The commit MUST stand — the email change is already successful on both sides — and the failure MUST be recorded as a `security_signal_failed` audit entry. The notification is NOT retried in the background.
- The post-commit acknowledgement notification to the new address fails to send (FR-016c). The commit MUST stand and the failure MUST be recorded as a `new_address_notification_failed` audit entry. Out-of-band admin contact is the recovery path.
- The post-commit global-admin fan-out publish fails (FR-016d). The commit MUST stand and the failure MUST be recorded as a `global_admin_notification_failed` audit entry. Note: this captures failure to PUBLISH the event; per-recipient delivery failures downstream of the notifications-service are out of this spec's scope.
- The post-commit session-invalidation call to the identity provider fails. The commit MUST stand and MUST NOT be rolled back; the failure MUST be recorded as a `session_invalidation_failed` audit entry. Previously-established sessions may continue until natural expiry or manual invalidation.

## Out of Scope

- **Email-ownership verification (any flow where the platform proves the new mailbox is controlled by the subject user before committing).** That flow is contracted in companion spec 098 (Self-Service Email Change). The admin is responsible for verifying the subject's identity out-of-band before invoking this spec's mutation.
- **Self-service email change initiated by the user themselves.** Also contracted in spec 098.
- Changing identity-provider-managed credentials other than the email trait (password, MFA factors, OIDC links).
- Changing the user's display name, contact preferences, or any non-login email fields. The Alkemio user has one login email; other addresses (if any) are not affected.
- Bulk admin email-change operations (migrating many users at once). Each change MUST go through the verified per-user authorization flow.
- A client-side UI for the flow. This specification covers the server-side capability; the client experience (admin console entry point) is handled in client-web and is referenced only where the server contract constrains it.
- Persisting any "pending" record. There is no multi-step lifecycle in this spec — the mutation commits in a single synchronous call. Spec 098 introduces a `pending` concept native to its multi-step verification flow.

## Requirements _(mandatory)_

### Functional Requirements

- **FR-002**: System MUST expose a server-side GraphQL mutation for a platform administrator to change the login email of any other user. The mutation MUST be placed under the `platformAdmin` surface, take a single `Input` object containing the subject user id and the new email, and commit the change synchronously (no multi-step "begin / confirm" lifecycle).
- **FR-002a**: The mutation MUST be available for any Kratos identity regardless of credential type (password credential, OIDC link, or both). The `email` trait change is applied identically in all cases; OIDC link bookkeeping is not modified. OIDC-only users MUST NOT be rejected on the basis that they lack a Kratos password credential.
- **FR-004**: System MUST validate the proposed new email's format and its uniqueness across both the Alkemio user records and the identity provider's identities before invoking any side-write.
- **FR-005**: System MUST reject a proposed new email that equals (case-insensitively) the subject user's current login email with a "no change" error.
- **FR-006**: System MUST treat email addresses case-insensitively for equality and conflict checks, while preserving the original casing supplied for display.
- **FR-009**: System MUST commit the email change such that, in effect, either both the Alkemio user record and the identity provider identity reflect the new email, or neither does. Before declaring a side write a failure, the platform MUST perform a bounded synchronous retry with backoff against the identity provider (2–3 attempts, total budget on the order of 5–10 seconds) to absorb transient errors. Once the retry budget is exhausted, the failure MUST be treated as terminal and trigger a compensating action that returns both sides to the pre-change state. Background / asynchronous reconciliation MUST NOT be used to defer commit beyond the synchronous budget.
- **FR-009a**: The compensating rollback (FR-009) MUST itself observe the same bounded synchronous retry budget (2–3 attempts, total budget ~5–10 seconds) when calling the failing side. If the rollback retry budget is exhausted without restoring the pre-change state, the system MUST: (a) record an audit entry with outcome `drift_detected` capturing the observed value on each side and a non-leaky failure reason, (b) surface a distinct error to the admin clearly identifying that drift remains and that admin-driven reconciliation is required (per FR-009b), and (c) emit an operations-alert signal via the existing observability channels — specifically, a Winston error-level log entry **and** an Elastic APM error capture, both tagged with a known event marker (e.g., `email_change_drift_detected`) so existing log-based and APM-based alerting can fire on it. No new ops-alert infrastructure is introduced by this feature. No asynchronous / background reconciler is introduced; reconciliation of a drift-detected case is admin-driven via the dedicated mutation defined in FR-009b.
- **FR-009b**: When a subject user has an outstanding drift-detected audit entry (most recent audit entry for that subject has outcome `drift_detected` and there is no subsequent `drift_resolved` entry), reconciliation MUST be performed via a dedicated `platformAdmin` GraphQL mutation introduced by this feature (`adminUserEmailChangeDriftResolve`), and NOT via direct PostgreSQL edits or direct Kratos admin-console operations. The mutation MUST take a single `Input` object identifying the subject user and the admin-chosen canonical email; the canonical email MUST be one of the two values observed on the drift-detected audit entry (i.e., the admin's choice is "keep the change" vs "revert the change"). The mutation MUST read current values on both sides, apply alignment writes only where needed, and observe the same bounded synchronous retry budget as FR-009 (2–3 attempts, total ~5–10 seconds per side that requires a write). On successful alignment, an audit entry with outcome `drift_resolved` MUST be written capturing actor, subject, the canonical email selected (recorded in the `new_email` column), the originating drift entry's old-side value (mirrored into `old_email` for forensic context), and timestamp — see data-model.md §Outcome semantics. Reading the audit row alone MUST be sufficient to identify the admin's direction (keep vs revert) without cross-referencing the originating `drift_detected` entry. On failure to align within the retry budget, the mutation MUST surface a distinct error and write an audit entry with outcome `drift_resolution_failed`. The original `drift_detected` audit entry MUST NOT be modified — it remains as forensic evidence per FR-014a.
- **FR-010**: System MUST update the identity provider's existing identity in place (same identity id). The Alkemio user's profile id and identity provider identity id MUST NOT change as a result of the email change; memberships, contributions, and historical references MUST remain associated with the same user.
- **FR-011**: System MUST mark the new email as verified on the identity provider side at commit time, so that the user is not prompted to re-verify the address after the change. (The admin's out-of-band manual verification is the trust source for this `verified=true` flag.)
- **FR-012**: After a successful change, the system MUST allow login with the new email and MUST reject login attempts using the old email.
- **FR-013**: System MUST authorize the email-change mutation such that only a platform administrator MAY invoke it for any user. Other actors MUST be rejected with an authorization error. (Self-service authorization — where a user initiates a change for themselves — is contracted in spec 098 and requires the platform-mediated email-ownership verification specified there.)
- **FR-014**: System MUST record an audit entry for every change attempt, capturing initiator, subject, old address, new address, timestamp, outcome (one of: `committed`, `rolled_back`, `drift_detected`, `drift_resolved`, `drift_resolution_failed`, `security_signal_failed`, `session_invalidation_failed`, `new_address_notification_failed`, `global_admin_notification_failed`, `rejected_validation`, `rejected_conflict`), and (where applicable) a non-leaky failure reason. Audit entries MUST NOT leak the existence of conflicting accounts in their failure-reason fields.
- **FR-014a**: Audit entries MUST be persisted in a dedicated entity introduced by this feature (`email_change_audit_entry`) backed by PostgreSQL, so that they remain queryable over time. The codebase has no pre-existing general-purpose audit entity to extend, so this entity is new. The existing `activity-log` infrastructure (user-facing collaboration activity feed) MUST NOT be reused for this purpose. Structured Winston log entries MAY be emitted in parallel for operational visibility but MUST NOT be treated as the system of record; the dedicated entity is the system of record. Audit entries MUST be retained indefinitely; this feature MUST NOT introduce an automatic purge or retention window for `email_change_audit_entry`. Removal of audit entries for a deleted subject user is delegated to the platform's existing user-deletion workflow and is out of scope here.
- **FR-014b**: System MUST expose a paginated GraphQL query under the `platformAdmin` surface that returns the `email_change_audit_entry` records for a given subject user, ordered by timestamp descending. The query MUST be keyed by subject user only; additional filters (by initiator, outcome, or date range) are out of scope for this iteration. The query MUST follow the project's pagination conventions (`docs/Pagination.md`). Returned fields MUST include: (a) `initiator` as a minimal user profile summary `{ id, displayName }`, (b) `initiatorRole` as a tag (`self` vs `platform admin`) distinguishing self-initiated (when spec 098 lands) from admin-on-behalf entries — this spec only writes `platform admin`, (c) `subject` as a minimal user profile summary `{ id, displayName }`, (d) old address, (e) new address, (f) timestamp, (g) outcome (from the audit-outcome enumeration introduced by this feature), and (h) where applicable, the non-leaky failure reason. The query MUST NOT expose the initiator's or subject's email address or any other PII beyond `id` and `displayName`. For entries whose `initiator` cannot be resolved to a profile summary, the planning artifact MUST define a sentinel representation; the `initiatorRole` tag MUST always be present. The `initiator_role` enum is contracted with both `self` and `platform_admin` values upfront so that no enum migration is required when spec 098 lands; until then, entries written by this spec carry `platform_admin` exclusively.
- **FR-015**: System MUST surface clear, distinct errors for each failure mode (identity provider unreachable, identity provider write failed, Alkemio write failed, conflict, invalid format, "no change", subject not found, drift detected, drift resolution failed, unauthorized) without leaking sensitive internals.
- **FR-016**: System MUST send a security-signal notification to the **old** email address after — and only after — an email change is committed. Rejected change attempts MUST NOT result in any message to the old address. The notification MUST contain: (a) a statement that the user's login email was changed, (b) the commit timestamp, (c) the initiator role (`platform admin` for this spec; `self` is added by spec 098), (d) a partially masked rendering of the new address (e.g., `j***@e***.com`) — the full new address MUST NOT be disclosed in this notification, and (e) recovery instructions for the user to follow if the change was not expected. The notification MUST NOT include the identity provider identity id or other internal identifiers.
- **FR-016a**: The security-signal notification MUST be rendered in English. The feature MUST NOT introduce per-user locale resolution, Accept-Language handling, or i18n template lookup for this message.
- **FR-016b**: If sending the post-commit security-signal notification (FR-016) fails, the platform MUST perform a bounded synchronous retry with backoff (2–3 attempts, total budget ~5–10 seconds) against the outbound mail channel. If the retry budget is exhausted without a successful send, the platform MUST write an audit entry with outcome `security_signal_failed` capturing subject, old address, commit timestamp, and the underlying failure reason in the non-leaky failure-reason field. The commit MUST stand — the email change is already successful on both sides at this point, and the security-signal failure MUST NOT trigger the FR-009 compensating rollback. No background / asynchronous retry of the security-signal notification is introduced; reconciliation in this case is out-of-band.
- **FR-016c**: System MUST send a post-commit acknowledgement notification to the **new** email address informing the new-mailbox holder that the address is now the user's login email on the platform. This is a distinct event from the FR-016 security signal at the OLD address — the two notifications have different recipients, different content rules, and different threat models. The notification MUST contain: (a) a statement that this address is now the user's login email on Alkemio, (b) the commit timestamp, (c) the initiator role (`platform admin` for this spec; `self` is added by spec 098), (d) a login link / instruction so the recipient can use the platform, and (e) a recovery / disclaimer line asking the recipient to contact support if the change was not expected. Unlike FR-016 (which masks the new email to avoid leaking the user's new address to a potentially-hostile recipient at the old address), FR-016c is addressed to the legitimate new-mailbox holder and MAY render the full new email. The notification MUST NOT include the identity-provider identity id or other internal identifiers. The notification MUST be rendered in English per FR-016a. If sending fails, the platform MUST perform a bounded synchronous retry with backoff (2–3 attempts, total budget ~5–10 seconds) against the outbound mail channel. If the retry budget is exhausted, the platform MUST write an audit entry with outcome `new_address_notification_failed` capturing subject, new address, commit timestamp, and the underlying failure reason in the non-leaky failure-reason field. The commit MUST stand — the email change is already successful on both sides at this point, and this failure MUST NOT trigger the FR-009 compensating rollback. No background / asynchronous retry is introduced; out-of-band reconciliation (e.g., admin contact) is the recovery path. Rejected change attempts MUST NOT result in any message to the new address.
- **FR-016d**: System MUST publish a post-commit notification event for delivery to **all platform / global administrators**, mirroring the existing platform "Global Role Change" notification pattern. The event MUST be published once via the existing notifications fan-out infrastructure (`NotificationExternalAdapter`); the server MUST NOT enumerate the recipient list at the server layer — recipient resolution and per-recipient channel selection (email, push / PWA, in-app) are responsibilities of the existing notifications service, identical to the Global Role Change flow. The notification payload MUST contain: (a) the subject user as a `{id, displayName}` profile summary, (b) the old email (full — admins already have access to this via the user-admin surface; the FR-016 anti-disclosure rationale does NOT apply because the recipients are trusted platform administrators), (c) the new email (full, same rationale), (d) the initiator as a `{id, displayName}` profile summary, (e) the initiator role tag (`platform admin` for this spec; `self` is added by spec 098), and (f) the commit timestamp. The notification payload MUST NOT include any other PII beyond the four `{id, displayName}` profile summaries and the two email addresses, MUST NOT include the identity-provider identity id, and MUST be rendered in English per FR-016a. The notification MUST fire on `committed` outcomes AND on `drift_detected` outcomes (the second case so other administrators are alerted to the drift state and can assist with reconciliation per FR-009b). On every other outcome (`drift_resolved`, `drift_resolution_failed`, `rolled_back`, `rejected_*`, `security_signal_failed`, `session_invalidation_failed`, `new_address_notification_failed`, `global_admin_notification_failed`) the notification MUST NOT fire. Specifically: `drift_resolved` does NOT trigger a fan-out — admins observe drift resolution via the audit query (FR-014b), and the original `drift_detected` fan-out already alerted them to the situation; firing again on resolution would be noise. On `drift_detected` the payload's `oldEmail` carries the Alkemio-side observed value and `newEmail` carries the Kratos-side observed value (mirroring the `drift_detected` audit row's column semantics per data-model.md §Outcome semantics); on `committed` the payload's `oldEmail` is the pre-change canonical and `newEmail` is the post-change canonical. If publish fails, the platform MUST perform a bounded synchronous retry with backoff (2–3 attempts, total budget ~5–10 seconds) against `NotificationExternalAdapter`. If the retry budget is exhausted, the platform MUST write an audit entry with outcome `global_admin_notification_failed` capturing subject, the relevant addresses, commit timestamp, and the underlying failure reason. The commit MUST stand. No background / asynchronous retry is introduced; out-of-band reconciliation is the recovery path. (Note: downstream delivery from the notifications-service to individual admin recipients across email / push / in-app channels is OUT OF SCOPE for this spec — the server's contract ends at the successful publish; any per-recipient delivery failure is observed and handled by the notifications service.)
- **FR-017**: System MUST invalidate every existing identity-provider session for the subject user when an email change is committed. After a commit, every device that was previously logged in as the user MUST be required to log in again, using the new email.
- **FR-017a**: If the identity provider's session-invalidation call fails after a successful two-side commit, the platform MUST perform a bounded synchronous retry with backoff (2–3 attempts, total budget ~5–10 seconds). If the retry budget is exhausted without confirmed invalidation, the platform MUST write an audit entry with outcome `session_invalidation_failed`. The commit MUST stand and MUST NOT be rolled back. The residual risk (previously-established sessions surviving until natural expiry or manual invalidation) is an audited degradation of SC-007 in this failure mode.
- **FR-018**: System MUST follow the project's GraphQL conventions: the admin email-change mutation is exposed via a mutation taking a single `Input` object, with a description, placed under the `platformAdmin` surface, with imperative naming. The drift-resolve mutation (FR-009b) follows the same conventions. (The self-service mutation in spec 098 is placed under the `me` surface, and 098's session-less confirm mutation is placed at the root.)
- **FR-021**: System MUST expose a GraphQL query under the `platformAdmin` surface that returns the most recent audit entry for a given subject user, so an admin can quickly check whether the subject is currently in a drift-detected state without paging through the audit history (FR-014b). The query returns null if no audit entry exists for the subject. Returned fields MUST be a subset of FR-014b's fields (no PII beyond `{id, displayName}` for initiator/subject).

### Key Entities

This feature introduces **one** entity:

- **Audit Entry** (`email_change_audit_entry`): An append-only record describing one step in the email-change lifecycle (commit, rollback, drift detection, drift resolution, drift-resolution failure, security-signal failure, session-invalidation failure, validation rejection, conflict rejection) with initiator, subject, old address, new address, timestamp, outcome, and a non-leaky failure reason where applicable. Persisted in PostgreSQL. Retained indefinitely (no purge by this feature; per-subject removal is delegated to the platform user-deletion workflow). Independent of any pre-existing audit / activity-log infrastructure. For drift-detected entries, `old_email` records the value observed on the Alkemio side at the moment of drift and `new_email` records the value observed on the identity-provider side (consistent with the Kratos-first commit ordering: at drift, Alkemio holds its old value because that write never started, and Kratos holds the new value because its write succeeded but the revert failed).

**Note**: This spec does NOT introduce a `pending` entity. There is no multi-step lifecycle; the admin mutation is a single synchronous call. Spec 098 introduces `email_change_pending` natively as part of its verification flow.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: A platform administrator can complete an admin-on-behalf email change end to end in a single synchronous mutation without any direct database edits, and the action is fully reflected in the audit trail with outcome = `committed`.
- **SC-002**: Across all completed email changes, zero observed cases of divergence between the Alkemio user record and the identity provider identity (i.e., zero cases where one side reflects the new email and the other does not), excluding cases explicitly recorded as `drift_detected` (which are the result of rollback-budget exhaustion and require admin reconciliation per FR-009b).
- **SC-003**: For every email change attempt — successful, rolled-back, drift-detected, drift-resolved, drift-resolution-failed, validation-rejected, conflict-rejected — the platform produces an audit entry within the same operation; spot-checking 100% of attempts in a test run yields an audit entry for each.
- **SC-004**: Failed second-side commits in a fault-injection test produce zero residual changes on either side and surface a clear, distinct error to the admin.
- **SC-005**: A double-failure (forward Kratos write fails AND the Kratos revert fails) produces a `drift_detected` audit entry capturing the observed value on each side, a Winston error log entry tagged `email_change_drift_detected`, and an Elastic APM `captureError` invocation with the same marker.
- **SC-006**: Validation errors (format, uniqueness, no-change) are returned without invoking any side-write; the audit entry records the rejection outcome and the database state is unchanged.
- **SC-007**: After a committed change, login attempts with the old email are rejected and login attempts with the new email succeed; every previously-active identity-provider session for that user is no longer accepted on its next request (subject to the FR-017a residual-risk caveat for session-invalidation failures).
- **SC-008**: After a committed change, the subject user's old email address receives exactly one security-signal notification (or, on FR-016b mail-send failure, zero notifications plus a `security_signal_failed` audit entry); rejected change attempts result in zero notifications to the old address.
- **SC-009**: After a committed change, the subject user's new email address receives exactly one acknowledgement notification (FR-016c) — or, on FR-016c mail-send failure, zero notifications plus a `new_address_notification_failed` audit entry. Rejected change attempts result in zero notifications to the new address.
- **SC-010**: A `committed` outcome publishes exactly one global-admin-fan-out event (FR-016d). A `drift_detected` outcome publishes exactly one. Every other outcome (`drift_resolved`, `drift_resolution_failed`, `rolled_back`, `rejected_*`, `*_failed`) publishes zero. On publish-budget exhaustion for either triggering outcome, zero events are published and one `global_admin_notification_failed` audit entry is written. Per-recipient downstream delivery to admin users (email / push / in-app) is observed by the notifications service and is out of this spec's measurement scope.

## Clarifications

### Session 2026-05-13 (original — preserved where still applicable)

- Q: When should the platform notify the user's old email address that their login email is being changed? → A: After commit only — send the security-signal notification once the change has been committed; rejected change attempts produce no notification.
- Q: What should happen to the user's existing identity-provider sessions when an email change is committed? → A: Invalidate every existing session for that user on commit; the user must log in again everywhere with the new email.
- Q: Should the email-change mutation enforce application-level rate limiting? → A: No application-level rate limit — rate limiting is delegated entirely to edge infrastructure (Oathkeeper / Kratos / upstream throttling).
- Q: How does a platform administrator learn the outcome of an admin-initiated email change? → A: The mutation is synchronous and returns the outcome directly; for forensic queries, the admin uses the audit-history query (FR-014b) and the latest-entry query (FR-021).
- Q: When the identity-provider write fails during commit, should the platform retry before rolling back? → A: Bounded synchronous retry with backoff (2–3 attempts, total budget ~5–10 seconds) before rolling back. Beyond the budget, the failure is treated as terminal and the existing compensating-rollback path runs unchanged. No asynchronous / background reconciliation.
- Q: What locale should be used for the security-signal notification? → A: Always English. No localization logic, locale lookup, or per-user locale resolution is introduced by this feature.
- Q: What information must the post-commit security-signal notification to the old email contain? → A: Statement of change, commit timestamp, initiator role (`platform admin` for this spec), partially masked new address (e.g., `j***@e***.com`), and recovery instructions. The full new address and internal identifiers MUST NOT be disclosed.
- Q: What happens when the compensating rollback itself fails after its retry budget is exhausted? → A: The rollback observes the same bounded retry budget as the commit (2–3 attempts, ~5–10 s). On exhaustion the system writes a `drift_detected` audit entry (capturing observed per-side values and the failure reason), surfaces a distinct error to the admin, and emits Winston + APM ops signals. No asynchronous reconciler is introduced; reconciliation is admin-driven via FR-009b.
- Q: Are users who authenticate exclusively via OIDC (no Kratos password credential) in scope for the admin email-change feature? → A: In scope, treated identically. All Kratos identities are eligible regardless of credential type; the email-trait change is applied the same way; OIDC link bookkeeping is unchanged.
- Q: By what mechanism is the FR-009a "operations-alert signal" emitted when drift is detected? → A: Existing observability channels only — a Winston error-level log entry and an Elastic APM error capture, both tagged with `email_change_drift_detected`. No new ops-alert infrastructure.
- Q: Where do audit entries live? → A: In a dedicated `email_change_audit_entry` entity introduced by this feature (PostgreSQL-backed). The existing `activity-log` infrastructure MUST NOT be reused. Winston log entries may be emitted in parallel for ops visibility but are not the system of record.
- Q: How long should `email_change_audit_entry` records be retained? → A: Indefinitely. This feature introduces no automatic purge for audit entries; GDPR right-to-erasure for a specific subject is handled by the existing user-deletion workflow.
- Q: Are `email_change_audit_entry` records queryable via GraphQL, and at what surface? → A: Yes. A paginated `platformAdmin` GraphQL query keyed by subject user returns that subject's audit entries (FR-014b), and a lightweight latest-entry query (FR-021) returns the most recent entry for quick drift checks. No `me`-shaped audit query is introduced in this spec.
- Q: By what mechanism does an admin reconcile a drift-detected change? → A: A dedicated `platformAdmin` GraphQL mutation introduced by this feature (FR-009b) is the only supported recovery path. It accepts the subject user and an admin-chosen canonical email, force-aligns both sides within the FR-009 retry budget, and writes a `drift_resolved` audit entry. Direct PostgreSQL edits and direct Kratos admin-console edits MUST NOT be used.
- Q: How should the platform handle a failure to send the post-commit security-signal notification to the old email? → A: Bounded synchronous retry (2–3 attempts, ~5–10s budget) against the mail channel, then on exhaustion write an audit entry with outcome `security_signal_failed` and continue — the commit stands. No background retry is introduced.
- Q: How should the platform handle a failure to invalidate existing identity-provider sessions on commit? → A: Bounded synchronous retry (2–3 attempts, ~5–10s budget), then on exhaustion write an audit entry with outcome `session_invalidation_failed` and continue — the commit stands and is not rolled back.
- Q: What shape do the `initiator` and `subject` fields take in the audit-entries query response? → A: Minimal user profile summary `{ id, displayName }` for both, plus a separate `initiatorRole` tag (`self` vs `platform admin`). Email addresses and other PII MUST NOT be returned.

### Session 2026-05-18 (spec split + smaller MVP)

- Q: Should the unified spec be split into separate admin-on-behalf and self-service specs? → A: Yes. Admin ships first; self-service is moved to companion spec 098 and depends on this spec's audit foundation.
- Q: Should the admin path require email-ownership verification of the new mailbox before committing? → A: **No** — the admin is expected to verify the subject's identity out-of-band (support call, ID check, prior account context). This spec is the smaller MVP: the admin mutation commits synchronously without sending a confirmation message and without requiring the new mailbox to prove ownership. The full platform-mediated email-ownership verification path is contracted in companion spec 098 (Self-Service Email Change), where no admin is in the loop and ownership of the new mailbox MUST be proven by the platform.
- Q: Without verification, what trust source authorises marking the new email as verified on the identity-provider side (FR-011)? → A: The admin's authorization combined with their out-of-band manual verification. The admin is accountable in the audit trail (FR-014, FR-014b) and the subject still receives a post-commit security-signal at the old address (FR-016) giving them the opportunity to detect and escalate an erroneous change.
- Q: Without verification, does this spec still need an `email_change_pending` entity, token machinery, or a multi-step state lifecycle? → A: No. The mutation is synchronous and atomic in one resolver call. The audit entry is the only persistent record. Drift state (when the compensating rollback also fails) lives on a `drift_detected` audit entry, not on a separate pending row. Spec 098 introduces the `email_change_pending` entity, the token, and the multi-step lifecycle natively as part of its verification flow.

## Assumptions

- The Alkemio user is already linked to a unique identity-provider identity id stored on the user record (stated in the issue: "needs the user's Kratos identity id, which we already store on the Alkemio user").
- The platform already operates an outbound mail capability suitable for sending the post-commit security-signal notification.
- "Platform administrator" for the purpose of this feature means an actor with platform-admin privilege as defined by the existing authorization model. Community / space administrators do NOT inherit this capability, consistent with the project's `platformAdmin` GraphQL placement convention.
- The admin performs identity verification of the subject user out-of-band (e.g., support call, ID check, prior account context) before invoking the mutation. This spec does NOT enforce or replace that out-of-band process; it relies on the admin's authorization as the sole platform-side gate.
- Email comparison for equality and uniqueness is case-insensitive on the local part as well as the domain. (This is permissive relative to RFC 5321, but aligns with how mainstream mail providers behave and with the identity provider's own behavior on the `email` trait.)
- "Atomic in effect" does not require a distributed transaction. A compensating-action approach is acceptable as long as the externally observable outcome matches the all-or-nothing requirement and the audit trail captures any compensations performed.
- Rate limiting for the mutation is provided by the existing edge infrastructure (Oathkeeper / Kratos / upstream throttling); the feature itself does not introduce per-user application-level counters or cooldowns.
- The `email_change_initiator_role` enum is shipped with both `self` and `platform_admin` values upfront, even though this spec only writes `platform_admin`, so that spec 098 introduces no enum migration when it lands.
- The platform already operates a notifications service with the existing "Global Role Change" fan-out pattern — a single server-side publish to the notifications adapter is fanned out by the service to all platform admins across the email / push / in-app channels. FR-016d reuses this pattern; the server does NOT enumerate admin recipients or select per-recipient channels. (See research.md §R8 for the adapter contract.)

## Dependencies

- The identity provider's admin API supports replacing an identity's `email` trait and the verified-address bookkeeping for an existing identity (i.e., updating in place by identity id), as documented in the issue's "Kratos behavior" section.
- An outbound mail channel is available for the post-commit security-signal notification.
- Existing authorization and audit infrastructure can be extended to cover the new operations.
