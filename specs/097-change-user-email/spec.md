# Feature Specification: Change User Login Email With Ownership Verification

**Feature Branch**: `097-change-user-email`
**Created**: 2026-05-13
**Status**: Draft
**Input**: User description: "implement https://github.com/orgs/alkem-io/projects/50/views/8?pane=issue&itemId=187119757&issue=alkem-io%7Cserver%7C6064"

**Related Issues**:

- alkem-io/server#6064 — Allow changing a user's login email while preserving their Alkemio profile

## User Scenarios & Testing _(mandatory)_

### User Story 1 — User Changes Their Own Login Email (Priority: P1)

As a registered Alkemio user, I want to change the email address I use to log in so that I can move authentication to a new mailbox (e.g., after a job change or provider migration) without losing my profile, memberships, contributions, or history.

**Why this priority**: This is the primary user-facing capability the feature exists to deliver. Without it, users have no self-service path to update their login email, and the platform falls back to ad-hoc database edits that have already drifted out of sync with the identity provider.

**Independent Test**: A logged-in user requests an email change to a new address, confirms ownership via the email sent to that new address, and is then able to log in only with the new address — the old address no longer authenticates. The user's profile id, memberships, and content remain unchanged.

**Acceptance Scenarios**:

1. **Given** a logged-in user with a valid login email, **When** the user requests to change their email to a new, unused address, **Then** the platform sends a confirmation message to the new address and records a pending change without yet altering the login email.
2. **Given** a pending email change with an unexpired confirmation token, **When** the user proves ownership of the new address (clicks the confirmation link in the message sent to that address), **Then** the login email is updated in both the Alkemio user record and the identity provider, the user's profile id is unchanged, and the pending change is marked complete.
3. **Given** the user has completed the change, **When** they attempt to log in with the new address, **Then** authentication succeeds; **When** they attempt to log in with the old address, **Then** authentication is rejected.
4. **Given** the user has completed the change, **When** they view their profile, **Then** all memberships, contributions, and historical references are intact and continue to reference the same user identity.

---

### User Story 2 — Platform Admin Changes a User's Login Email On Their Behalf (Priority: P1)

As a platform administrator, I want to initiate an email change for another user (e.g., when the user has lost access to their old mailbox and contacts support) so that I can restore their access while still requiring proof that they control the new mailbox.

**Why this priority**: Admin-on-behalf-of is a primary support flow. Many email-change requests arrive precisely because the user can no longer access the old address; the user therefore cannot initiate a self-service flow from within Alkemio. Without this path, support has no way to resolve the case except by editing the database, which is exactly what this feature replaces.

**Independent Test**: A platform admin initiates an email change for another user's account, the confirmation message is sent to the proposed new address, and once the target user confirms ownership, the change is committed across both systems. The action is recorded in the audit trail with the admin's identity, the target user's identity, the old address, and the new address.

**Acceptance Scenarios**:

1. **Given** a platform admin acting on behalf of another user, **When** the admin initiates an email change for that user with a new address, **Then** the platform validates the new address (format + uniqueness) and sends a confirmation message to it, recording a pending change attributed to the admin.
2. **Given** the pending admin-initiated change, **When** the target user confirms ownership via the confirmation message, **Then** the change is committed across the Alkemio user record and the identity provider, and an audit entry is recorded that captures the actor (admin), the subject (user), the old address, and the new address.
3. **Given** a non-admin user, **When** they attempt to initiate an email change for someone else, **Then** the platform rejects the request with an authorization error and no message is sent.

---

### User Story 3 — Atomic Two-Side Commit With Rollback on Partial Failure (Priority: P1)

As a platform operator, I want the email change to either fully succeed across both Alkemio and the identity provider or fully fail with no partial state, so that the two systems cannot silently drift out of sync the way they did under the legacy database-edit approach.

**Why this priority**: Drift between the Alkemio user record and the identity provider is the root failure mode the issue exists to fix. Even a perfect verification flow is unsafe if a mid-commit failure leaves Alkemio updated but the identity provider unchanged (or vice versa). Without this guarantee, the feature reintroduces the very problem it was created to solve.

**Independent Test**: Simulate a failure on the second-side write (e.g., the identity provider is unreachable or returns an error) after the first side has been updated, and verify that the first-side change is rolled back so the user's login email is unchanged on both sides. A clear error is surfaced to the initiator.

**Acceptance Scenarios**:

1. **Given** a confirmed pending change, **When** the platform updates the Alkemio user record successfully but the identity provider write fails, **Then** the platform reverses the Alkemio update so both sides reflect the old email, and surfaces a clear error to the initiator.
2. **Given** a confirmed pending change, **When** the identity provider write succeeds but persisting the Alkemio user record fails, **Then** the platform reverses the identity provider update so both sides reflect the old email, and surfaces a clear error to the initiator.
3. **Given** any partial-failure recovery has run, **When** the user next attempts to log in, **Then** the old email continues to work (the system is in its pre-change state), and the user can retry the change.

---

### User Story 4 — Confirmation Token Lifecycle (Priority: P2)

As a security-conscious platform, I want confirmation tokens to be short-lived, single-use, and bound to the specific (user, proposed new email) pair, so that a leaked or replayed token cannot be used to hijack an account or to redirect a different user's email change.

**Why this priority**: The verification token is the only thing standing between an initiator and a committed email change. Without strict lifecycle rules, leaked links and stale tokens become a credible attack surface. P2 because the underlying flow (P1 stories) is what the user sees; this story is what makes the flow safe to operate.

**Independent Test**: Issue a confirmation token, then attempt to use it (a) twice, (b) for a different user, (c) after expiry, and (d) after the user has initiated a newer change. Each MUST be rejected with a clear error, and no email change MUST be committed.

**Acceptance Scenarios**:

1. **Given** an unused confirmation token, **When** the user confirms ownership before the token expires, **Then** the change proceeds and the token is marked consumed.
2. **Given** a confirmation token that has already been consumed, **When** the same token is presented again, **Then** the platform rejects it and surfaces a clear "already used" error.
3. **Given** a confirmation token that has expired, **When** the user attempts to confirm, **Then** the platform rejects it and surfaces a clear "expired" error, optionally inviting the user to restart the change.
4. **Given** an outstanding pending change for a user, **When** the same user (or admin) initiates a new email change for that user, **Then** the previous token is invalidated and only the new token is honored.

---

### User Story 5 — Conflict and Validation Errors Before Sending Mail (Priority: P2)

As any initiator of an email change, I want to be told immediately when the proposed new address is invalid or already in use, so that no confirmation mail is sent to an address that could never be assigned.

**Why this priority**: Validation before sending mail prevents accidental disclosure (sending a confirmation to an address that belongs to a different user), reduces spam complaints, and gives the initiator immediate feedback. P2 because the system would still be correct if uniqueness were only enforced on commit, but the user experience and abuse posture are materially worse.

**Independent Test**: Submit an email change with (a) a malformed address, (b) the user's current address, and (c) an address already belonging to another Alkemio user or identity provider identity. Each MUST be rejected before any mail is sent.

**Acceptance Scenarios**:

1. **Given** a malformed new email address, **When** the change is requested, **Then** the platform rejects with a validation error and does not send mail.
2. **Given** a new email identical to the user's current login email, **When** the change is requested, **Then** the platform rejects with a "no change" error and does not send mail.
3. **Given** a new email already in use by another Alkemio user or identity provider identity, **When** the change is requested, **Then** the platform rejects with a "conflict" error and does not send mail, and the error MUST NOT confirm or deny which user owns the conflicting address (to avoid enumeration).

---

### User Story 6 — Audit Trail of Email Changes (Priority: P3)

As a platform operator (and, for the user's own benefit, as the user themselves), I want every successful email change recorded with who initiated it, who the subject was, the old address, the new address, and when it happened, so that we can investigate incidents and answer "did anyone touch this account?" later.

**Why this priority**: Audit logging is critical for security investigations and for support, but the system is functional without it; therefore P3. It MUST land with the feature, not retrofitted.

**Independent Test**: After any successful email change (self- or admin-initiated), an audit entry is retrievable that includes initiator, subject, old address, new address, timestamp, and outcome. Failed attempts (rejected validation, expired token, partial-failure rollback) also produce entries with the failure reason.

**Acceptance Scenarios**:

1. **Given** a successfully committed email change, **When** the audit log is queried, **Then** an entry exists with initiator, subject, old address, new address, timestamp, and outcome = success.
2. **Given** a partial-failure rollback, **When** the audit log is queried, **Then** an entry exists describing the failure, the side(s) on which it occurred, and that both sides were restored.
3. **Given** a rejected attempt (validation, conflict, expired/used token), **When** the audit log is queried, **Then** an entry exists describing the rejection reason without leaking sensitive details (e.g., the existence of conflicting accounts).

---

### Edge Cases

- The user's old mailbox is already unreachable when the change is initiated. The self-service path requires login but not access to the old mailbox; the admin-on-behalf path explicitly supports this case.
- The user requests a change, then immediately requests another change to a different address before confirming the first. The system MUST invalidate the prior pending token and accept only the most recent one.
- The user clicks the confirmation link in a different browser / device than where they initiated the change. The confirmation MUST succeed regardless of session continuity, because the token is the proof of ownership; no session binding is required for the confirmation step itself.
- The user confirms but their Alkemio session has expired in the meantime. The confirmation MUST still commit the change (the token, not the session, is the authority for the confirmation step).
- The new email is identical (case-insensitive) to the current email. The platform MUST treat addresses case-insensitively for equality and conflict checks.
- The identity provider returns a non-error but unexpected response shape (e.g., the identity id no longer exists). The platform MUST treat this as a failure of the commit and trigger rollback rather than recording success.
- Two confirmations arrive simultaneously (e.g., the user clicks the link twice in quick succession). The platform MUST commit exactly once and reject the second attempt as "already used".
- The user is in a partial-failure rollback state when they retry. The retry MUST succeed normally, as the system was returned to its pre-change state.
- The pending change exists for a user whose Alkemio account is subsequently deactivated or deleted before confirmation. The token MUST be rejected on confirmation and the pending change cleaned up.

## Out of Scope

- Changing identity-provider-managed credentials other than the email trait (password, MFA factors, OIDC links).
- Changing the user's display name, contact preferences, or any non-login email fields. The Alkemio user has one login email; other addresses (if any) are not affected.
- Bulk admin email-change operations (migrating many users at once). Each change MUST go through the verified per-user flow.
- A client-side UI for the flow. This specification covers the server-side capability; the client experience (initiation form, confirmation landing page, admin console entry point) is handled in client-web and is referenced only where the server contract constrains it.

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: System MUST expose a server-side capability for a logged-in user to initiate a change to their own login email.
- **FR-002**: System MUST expose a server-side capability for a platform administrator to initiate a login-email change on behalf of any other user.
- **FR-002a**: The email-change capability (FR-001 and FR-002) MUST be available for any Kratos identity regardless of credential type (password credential, OIDC link, or both). The `email` trait change is applied identically in all cases; OIDC link bookkeeping is not modified (consistent with the existing Out of Scope declaration). OIDC-only users MUST NOT be rejected at initiation on the basis that they lack a Kratos password credential.
- **FR-003**: System MUST require proof of ownership of the proposed new email before committing the change on either side. Proof is established by the user (or, in the admin-on-behalf case, the target user) demonstrating control of the new mailbox via a confirmation message sent to that address.
- **FR-003a**: The confirmation message MUST contain a clickable link whose target carries an opaque single-use token. The link MUST target a client-web URL (the client application's deep link for confirming an email change), NOT a server-side HTTP endpoint; the client-web app reads the token from the URL and invokes the GraphQL confirm mutation. The server MUST NOT introduce a new HTTP route to consume the token directly; all confirmation logic remains behind the GraphQL surface. Confirmation MUST be reachable solely by following that link; the feature MUST NOT additionally emit a user-typeable code, and the confirm mutation MUST accept a single token string (no separate code-based code path).
- **FR-003b**: The confirmation message body MUST contain, in addition to the clickable link from FR-003a: (a) the initiator role tag (`self` vs `platform admin`) so the recipient understands whether the change was initiated by them or by an administrator acting on their behalf, (b) an explicit statement that the link is time-limited and consistent with the 1-hour TTL contracted in FR-007b (exact wording is an implementation choice, but the message MUST convey time-bound urgency rather than leaving expiry unstated), and (c) a recovery / disclaimer line instructing the recipient to ignore the message and contact support if the change was not expected. The message MUST NOT include: the initiating admin's display name, user id, or other admin PII (only the role tag is permitted — this is intentionally narrower than the FR-022 admin-profile disclosure, because the new-mailbox recipient may be a third party at the moment of read); the confirmation token outside the link itself; the identity provider identity id; or any other internal identifier. The message MUST be rendered in English per FR-016a; no per-user locale resolution is introduced. The trust-signal pattern (role tag + expiry + recovery instructions) intentionally mirrors FR-016's content contract for the post-commit security-signal notification at the old address.
- **FR-004**: System MUST validate the proposed new email's format and its uniqueness across both the Alkemio user records and the identity provider's identities before sending any confirmation message.
- **FR-004a**: System MUST re-validate the proposed new email's uniqueness at confirmation time, before any side-write to either Alkemio or the identity provider. The re-check applies the same scope as FR-004 (Alkemio user records and identity provider identities) and the same case-insensitivity rules from FR-006. If the proposed email is now in use by another Alkemio user or identity provider identity, the confirmation MUST be rejected with a `conflict` error (per FR-015), no side-write MUST occur on either side, and an audit entry with the `conflict` rejection outcome MUST be written (subject to the non-leaky failure-reason rules of FR-014 — the entry MUST NOT confirm or deny which user owns the conflicting address). The pending-change record MUST NOT transition to `confirmed` or `committed` in this case; supersession by a new initiation (per FR-007(d) and FR-019) remains the user's path forward. This narrows the window in which the FR-009 compensating-rollback path would otherwise be exercised for a benign uniqueness race between initiation and confirmation, while preserving the FR-009 guarantee for genuine commit-time failures.
- **FR-005**: System MUST reject a proposed new email that equals (case-insensitively) the user's current login email with a "no change" error, and MUST NOT send a confirmation message in that case.
- **FR-006**: System MUST treat email addresses case-insensitively for equality and conflict checks, while preserving the original casing supplied by the user for display.
- **FR-007**: System MUST issue confirmation tokens that are (a) short-lived, (b) single-use, (c) scoped to the specific user and the specific proposed new email, and (d) invalidated when a newer pending change is initiated for the same user.
- **FR-007a**: The confirmation token MUST be persisted as plaintext on the pending-change record (no separate hash column). Confirmation lookups compare the presented token against the stored plaintext value directly. The token's safety relies on (i) its short TTL (per FR-007(a) and the concrete value in FR-007b), (ii) its single-use semantics (FR-007(b)), (iii) the invalidation rule on supersession (FR-007(d)), (iv) database access controls applied to the pending-change record, and (v) its entropy floor (per FR-007c). Audit entries (FR-014/FR-014a) MUST NOT contain the token value.
- **FR-007c**: The confirmation token MUST be generated from a cryptographically secure random source and MUST carry at least 128 bits of entropy. The token's serialized form MUST be URL-safe (e.g., base64url or hex) so that it can be embedded as a query-string parameter in the client-web deep link defined in FR-003a without further encoding. The serialized form's character length is an implementation choice subject to the 128-bit floor (e.g., 22+ characters of base64url, or 32+ characters of hex). The token MUST NOT encode or derive from the subject user id, the proposed new email, the issue timestamp, or any other identifier — it is an opaque random value, and the binding to the (user, proposed new email) pair is established by the pending-change record, not by the token's internal structure. The 128-bit entropy floor is a contract surface; raising it (e.g., to 256 bits) is permissible without a spec change, but lowering it MUST be treated as a spec-level decision.
- **FR-007b**: The confirmation token's TTL MUST be exactly 1 hour from its issue timestamp. The 1-hour TTL applies uniformly to both self-initiated and admin-initiated changes — the platform MUST NOT introduce a tiered TTL keyed on initiator role. Tokens presented after the 1-hour boundary MUST be rejected per FR-007(a) and FR-008 and MUST surface the expired-token error per FR-015. The same 1-hour expiry MUST be the value materialized in the pending-change record's `expiryTimestamp` field returned by FR-022. The TTL is a contract surface; changing it is a spec-level decision, not an environment-level configuration.
- **FR-008**: System MUST consume the confirmation token atomically: a successful confirmation triggers the two-side commit; a failed or rejected confirmation MUST NOT alter the login email on either side.
- **FR-009**: System MUST commit the email change such that, in effect, either both the Alkemio user record and the identity provider identity reflect the new email, or neither does. Before declaring a side write a failure, the platform MUST perform a bounded synchronous retry with backoff against the identity provider (small number of attempts, total budget on the order of 5–10 seconds) to absorb transient errors. Once the retry budget is exhausted, the failure MUST be treated as terminal and trigger a compensating action that returns both sides to the pre-change state. Background / asynchronous reconciliation MUST NOT be used to defer commit beyond the synchronous budget.
- **FR-009a**: The compensating rollback (FR-009) MUST itself observe the same bounded synchronous retry budget (2–3 attempts, total budget ~5–10 seconds) when calling the failing side. If the rollback retry budget is exhausted without restoring the pre-change state, the system MUST: (a) transition the pending change to the terminal state `drift_detected` (a seventh state added to the FR-021 enumeration), (b) write an audit entry with outcome `drift_detected` capturing the observed values on each side and the failure reason, (c) surface a distinct error to the initiator clearly identifying that drift remains and that admin-driven reconciliation is required (per FR-009b), and (d) emit an operations-alert signal via the existing observability channels — specifically, a Winston error-level log entry **and** an Elastic APM error capture, both tagged with a known event marker (e.g., `email_change_drift_detected`) so existing log-based and APM-based alerting can fire on it. No new ops-alert infrastructure (e.g., dedicated RabbitMQ queue, webhook, or paging integration) is introduced by this feature. No asynchronous / background reconciler is introduced; reconciliation of a `drift_detected` pending change is an admin-driven synchronous operation invoked via the dedicated mutation defined in FR-009b, preserving the FR-009 guardrail against background reconciliation.
- **FR-009b**: When a pending email change is in the `drift_detected` terminal state (per FR-009a), reconciliation MUST be performed via a dedicated `platformAdmin` GraphQL mutation introduced by this feature (e.g., `platformAdminResolveEmailChangeDrift`), and NOT via direct PostgreSQL edits or direct Kratos admin-console operations. The mutation MUST take a single `Input` object identifying the subject user and the admin-chosen canonical email; the canonical email MUST be one of the two values observed on the `drift_detected` record (i.e., the admin's choice is "keep the change" vs "revert the change"). The mutation MUST read current values on both sides, apply alignment writes only where needed, and observe the same bounded synchronous retry budget as FR-009 (2–3 attempts, total ~5–10 seconds per side that requires a write). On successful alignment, an audit entry with outcome `drift_resolved` MUST be written capturing actor, subject, canonical email selected, prior observed values per side, and timestamp. On failure to align within the retry budget, the mutation MUST surface a distinct error and write an audit entry with outcome `drift_resolution_failed`; the pending-change record MUST remain in `drift_detected`. The `drift_detected` pending-change record itself MUST NOT transition state on successful resolution; it remains as forensic evidence (and is purged on the same 30-day window per FR-020, while the audit entries persist indefinitely per FR-014a).
- **FR-010**: System MUST update the identity provider's existing identity in place (same identity id). The Alkemio user's profile id and identity provider identity id MUST NOT change as a result of the email change; memberships, contributions, and historical references MUST remain associated with the same user.
- **FR-011**: System MUST mark the new email as verified on the identity provider side at commit time, so that the user is not prompted to re-verify the address after the change.
- **FR-012**: After a successful change, the system MUST allow login with the new email and MUST reject login attempts using the old email.
- **FR-013**: System MUST authorize the initiation endpoint such that a user MAY initiate a change only for themselves, and a platform administrator MAY initiate a change for any user. Other actors MUST be rejected with an authorization error.
- **FR-014**: System MUST record an audit entry for every initiation, every confirmation outcome (success, rejection, expiry), and every rollback, capturing initiator, subject, old address, new address, timestamp, and outcome. Audit entries MUST NOT leak the existence of conflicting accounts in their failure-reason fields.
- **FR-014a**: Audit entries MUST be persisted in a dedicated entity introduced by this feature (e.g., `email_change_audit_entry`) backed by PostgreSQL, so that they remain queryable beyond the 30-day pending-change retention window (FR-020) and so that events with no corresponding pending-change record (validation rejection before persistence; initiation rolled back under FR-019a) still have a durable home. The codebase has no pre-existing general-purpose audit entity to extend, so this entity is new. The existing `activity-log` infrastructure (user-facing collaboration activity feed) MUST NOT be reused for this purpose. Structured Winston log entries MAY be emitted in parallel for operational visibility but MUST NOT be treated as the system of record; the dedicated entity is the system of record per FR-020. Audit entries MUST be retained indefinitely; this feature MUST NOT introduce an automatic purge or retention window for `email_change_audit_entry`. Removal of audit entries for a deleted subject user is delegated to the platform's existing user-deletion workflow and is out of scope here.
- **FR-014b**: System MUST expose a paginated GraphQL query under the `platformAdmin` surface that returns the `email_change_audit_entry` records for a given subject user, ordered by timestamp descending. The query MUST be keyed by subject user only; additional filters (by initiator, outcome, or date range) are out of scope for this iteration. The query MUST follow the project's pagination conventions (`docs/Pagination.md`). Returned fields MUST include: (a) `initiator` as a minimal user profile summary `{ id, displayName }` (matching the FR-022 pattern), (b) `initiatorRole` as a tag (`self` vs `platform admin`) distinguishing self-initiated from admin-on-behalf entries, (c) `subject` as a minimal user profile summary `{ id, displayName }`, (d) old address, (e) new address, (f) timestamp, (g) outcome (from the audit-outcome enumeration introduced by this feature — `initiated`, `confirmed`, `committed`, `rolled_back`, `expired`, `superseded`, `drift_detected`, `drift_resolved`, `drift_resolution_failed`, `initiation_failed`, `security_signal_failed`, `session_invalidation_failed`, plus rejection outcomes for validation/conflict/used-token/expired-token presentations — the planning artifact MUST enumerate the final canonical set), and (h) where applicable, the non-leaky failure reason. The query MUST NOT expose the confirmation token; MUST NOT expose the initiator's or subject's email address or any other PII beyond `id` and `displayName`; MUST NOT return any field that has been redacted or masked elsewhere in the feature (the old/new addresses on the audit entry are the canonical addresses observed at the moment the entry was written, not the FR-016 masked form). For entries whose `initiator` cannot be resolved to a profile summary (e.g., validation-rejection entries written before the initiator's identity was fully resolved per FR-014a), the planning artifact MUST define a sentinel representation; the `initiatorRole` tag MUST always be present. No `me`-shaped audit query is introduced for the calling user in this iteration; users observe their own active pending change via FR-022 and the post-commit security-signal notification via FR-016.
- **FR-015**: System MUST surface clear, distinct errors for each failure mode (identity provider unreachable, identity provider write failed, Alkemio write failed, conflict, expired token, used token, invalid format, "no change") without leaking sensitive internals.
- **FR-016**: System MUST send a security-signal notification to the **old** email address after — and only after — an email change is committed. Abandoned or rejected change attempts MUST NOT result in any message to the old address. The notification MUST contain: (a) a statement that the user's login email was changed, (b) the commit timestamp, (c) the initiator role (`self` vs `platform admin`), (d) a partially masked rendering of the new address (e.g., `j***@e***.com`) — the full new address MUST NOT be disclosed in this notification, and (e) recovery instructions for the user to follow if the change was not expected. The notification MUST NOT include the confirmation token, the identity provider identity id, or other internal identifiers.
- **FR-016a**: Both the confirmation message (to the proposed new address) and the security-signal notification (to the old address) MUST be rendered in English. The feature MUST NOT introduce per-user locale resolution, Accept-Language handling, or i18n template lookup for these messages.
- **FR-016b**: If sending the post-commit security-signal notification (FR-016) fails, the platform MUST perform a bounded synchronous retry with backoff (2–3 attempts, total budget ~5–10 seconds) against the outbound mail channel. If the retry budget is exhausted without a successful send, the platform MUST write an audit entry with outcome `security_signal_failed` (a new audit outcome introduced by this requirement) capturing subject, old address (masked consistent with FR-016 — full old address MUST NOT appear in the audit entry beyond what is already permitted for other outcomes), commit timestamp, and the underlying failure reason in the non-leaky failure-reason field (per FR-014). The commit MUST stand — the email change is already successful on both sides at this point, and the security-signal failure MUST NOT trigger the FR-009 compensating rollback. No background / asynchronous retry of the security-signal notification is introduced; reconciliation in this case is out-of-band (e.g., admin-driven user contact). The post-commit email-change is otherwise unaffected: the user can log in with the new email and existing sessions are invalidated per FR-017.
- **FR-017**: System MUST invalidate every existing identity-provider session for the user when an email change is committed, regardless of which session (if any) initiated the change. After a commit, every device that was previously logged in as the user MUST be required to log in again, using the new email.
- **FR-017a**: If the identity provider's session-invalidation call fails after a successful two-side commit, the platform MUST perform a bounded synchronous retry with backoff (2–3 attempts, total budget ~5–10 seconds). If the retry budget is exhausted without confirmed invalidation of all existing sessions, the platform MUST write an audit entry with outcome `session_invalidation_failed` (a new audit outcome introduced by this requirement) capturing the underlying failure reason in the non-leaky failure-reason field (per FR-014). The commit MUST stand — the email change is already successful on both sides — and the session-invalidation failure MUST NOT trigger the FR-009 compensating rollback (rolling back the committed email change would itself introduce drift risk and would not improve session safety). No background / asynchronous retry of session invalidation is introduced; recovery is out-of-band (admin-driven via the existing identity-provider admin surface, not via a feature-owned mutation). Operationally, the residual risk is that previously-established sessions for the user may continue to validate against the identity provider until each session naturally expires or is invalidated manually; this is a known, audited degradation of SC-009 in this failure mode rather than a silent loss of guarantee.
- **FR-018**: System MUST follow the project's GraphQL conventions: the change is exposed via mutations taking a single `Input` object, with descriptions, with the self-service mutation placed under the `me`-shaped surface and the admin-on-behalf mutation placed under the `platformAdmin` surface, and with imperative naming.
- **FR-018a**: The confirm-email-change mutation MUST be exposed as a top-level (root) GraphQL mutation, callable without an Alkemio session. The token presented in the mutation input is the sole authority for the confirmation; the resolver MUST NOT require, infer, or rely on any caller session for authorization. Authorization is enforced solely by the pending-change token lookup: a presented token that matches an active pending-change record proceeds to the FR-004a uniqueness re-check and the FR-009 two-side commit path; any other presentation (no match, expired, already consumed, or superseded per FR-007(d)) is rejected with the corresponding FR-015 error surface and audited per FR-014. This placement is intentionally distinct from FR-018's `me` / `platformAdmin` placements for the two initiation mutations, because the edge-case contract requires confirmation to succeed regardless of session state — the caller's Alkemio session may be absent, expired, or belong to a different account at the moment of confirmation, and the token (not the session) is the authority.
- **FR-019**: System MUST be idempotent on retry of the initiation step in the sense that re-submitting the same proposed new email while a pending change exists does not create duplicate tokens; it MUST refresh the pending change and replace the token (in line with FR-007(d)).
- **FR-019a**: Initiation MUST be atomic across "persist pending change" and "send confirmation message": if the outbound mail provider returns a hard failure after FR-004 / FR-005 / FR-006 validation has passed, the just-persisted pending change MUST be rolled back so that no record remains, the confirmation token MUST be discarded (and treated as never issued), no audit entry of the `initiated` outcome MUST be retained on success — instead the initiation MUST be audited with an `initiation_failed` outcome describing the mail-send failure — and the initiator MUST receive a distinct error surface (per FR-015) identifying the failure as a mail-delivery problem. The initiator MAY retry immediately; a retry is a fresh initiation, not a refresh under FR-019.
- **FR-020**: System MUST retain pending-change records for 30 days after they reach a terminal state (`committed`, `rolled_back`, `expired`, `superseded`) and MUST remove them after that window so that operational state does not grow without bound. The raw audit log (FR-014) is the system of record for longer-term investigation and is not bounded by this 30-day window.
- **FR-021**: System MUST expose a GraphQL status query under the `platformAdmin` surface that returns the current lifecycle state of an email change for a given subject user, drawn from the set `initiated`, `confirmed`, `committed`, `rolled_back`, `expired`, `superseded`, `drift_detected`. The query MUST cover the active pending change (if any) and any pending-change records that reached a terminal state within the past 30 days (the audit window defined by FR-020), so that an admin who initiated a change on behalf of a user can observe the outcome without consulting the raw audit log. No in-app, push, or email notification to the admin is introduced.
- **FR-022**: System MUST expose a `me`-shaped GraphQL query that returns the calling user's active pending email change, if any, including the state (`initiated`, `confirmed`, or `drift_detected` — see below), the initiator role (`self` vs `platform admin`), the proposed new email, the issue timestamp, and the expiry timestamp. When the initiator role is `platform admin`, the response MUST additionally include a minimal admin profile summary consisting of the admin's user id and display name, so that the subject user can identify which administrator initiated the change and verify its legitimacy. The admin's email address and any other admin PII (contact details, roles beyond the profile summary, etc.) MUST NOT be exposed by this query. When the initiator role is `self`, no separate initiator profile is included (the caller already knows their own identity). The query MUST NOT return any pending change belonging to another user, and MUST NOT expose the confirmation token. No mutation is provided for the subject user to cancel a pending change; allowing the token to expire (or initiating a new change that supersedes the prior one) is the only user-side way to abandon it. No proactive notification is sent to the subject user when an admin-initiated change is created; they are expected to learn of it through the confirmation message at the new address, through this query, or post-commit through the security-signal notification at the old address. The admin-identity disclosure introduced here applies ONLY to FR-022; the post-commit security-signal notification (FR-016) remains role-only by design and is not modified.
- **FR-022a**: For the purposes of FR-022, "active" includes any pending change in state `initiated` or `confirmed`, AND any pending change in state `drift_detected` (terminal per FR-009a) for which the 30-day retention window (FR-020) has not yet elapsed. When the returned record is in state `drift_detected`, the response MUST additionally include a boolean indicator that admin reconciliation is required (e.g., `awaitingAdminReconciliation: true`), so that the subject user can identify the abnormal state and contact support / a platform admin. The response in the `drift_detected` case MUST NOT expose the per-side observed email values, the underlying failure reason in technical detail, or any other internal diagnostic — those remain visible only via the audit query (FR-014b) under the `platformAdmin` surface. Other terminal states (`committed`, `rolled_back`, `expired`, `superseded`) MUST NOT be returned by FR-022; the subject user observes those outcomes via the confirmation message, the post-commit security-signal notification (FR-016), or by attempting to log in.

### Key Entities

- **User (Alkemio)**: The platform-side account whose login email is being changed. Holds the canonical Alkemio user id (stable across the change), the current login email, and the link to the identity provider identity. Only the login email changes; identity, memberships, contributions, history are unaffected.
- **Identity Provider Identity**: The authentication-side record (linked to the Alkemio user by the user id) whose `email` trait determines what address can be used to log in. Same identity id before and after the change; the `email` trait and the verified-address bookkeeping are updated.
- **Pending Email Change**: A short-lived record representing an in-flight change request. Holds the subject user, the initiator (self or admin), the proposed new email, the issued confirmation token stored as plaintext (per FR-007a), the issue timestamp, the expiry timestamp, and the consumed/invalidated state. There is at most one active pending change per user; a new initiation replaces any prior pending change for the same user.
- **Confirmation Token**: The single-use proof-of-ownership credential sent to the proposed new email. Bound to a specific pending change (and therefore to a specific user and a specific proposed email). Cannot be transferred between users or pending changes.
- **Audit Entry**: An append-only record describing one step in the email-change lifecycle (initiate, confirm, reject, commit, rollback) with initiator, subject, old address, new address, timestamp, outcome, and a non-leaky failure reason where applicable. Persisted in a dedicated `email_change_audit_entry` entity introduced by this feature (PostgreSQL-backed). Lives beyond the 30-day pending-change retention window (FR-020) and is not co-located with the pending-change record. Retained indefinitely (no purge by this feature; per-subject removal is delegated to the platform user-deletion workflow). Independent of any pre-existing audit / activity-log infrastructure.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: A user can complete a self-service email change end to end (initiate → confirm → log in with new email) in a single sitting, with no support intervention, and without the platform losing any of the user's existing memberships, contributions, or history.
- **SC-002**: A platform administrator can complete an admin-on-behalf email change end to end for any other user (initiate → target user confirms → admin observes the outcome via the `platformAdmin` status query) without any direct database edits, and the action is fully reflected in the audit trail.
- **SC-003**: Across all completed email changes, zero observed cases of divergence between the Alkemio user record and the identity provider identity (i.e., zero cases where one side reflects the new email and the other does not).
- **SC-004**: For every email change attempt, the platform produces an audit entry within the same operation; spot-checking 100% of attempts in a test run yields an audit entry for each.
- **SC-005**: Failed second-side commits in a fault-injection test produce zero residual changes on either side and surface a clear, distinct error to the initiator.
- **SC-006**: Confirmation tokens cannot be reused, transferred to a different user, or accepted after expiry, as verified by adversarial test cases.
- **SC-007**: Validation errors (format, uniqueness, no-change) are returned before any confirmation mail is sent; zero confirmation mails are sent for rejected attempts in a test run.
- **SC-008**: After a committed change, login attempts with the old email are rejected 100% of the time, and login attempts with the new email succeed for a valid credential.
- **SC-009**: After a committed change, every previously-active identity-provider session for that user is no longer accepted on its next request; sessions cannot survive an email change.
- **SC-010**: After a committed change, the user's old email address receives exactly one security-signal notification; abandoned or rejected change attempts result in zero notifications to the old address.

## Clarifications

### Session 2026-05-13

- Q: When should the platform notify the user's old email address that their login email is being changed? → A: After commit only — send the security-signal notification once the change has been committed; abandoned or rejected change attempts produce no notification.
- Q: What should happen to the user's existing identity-provider sessions when an email change is committed? → A: Invalidate every existing session for that user on commit, regardless of which session (if any) initiated the change; the user must log in again everywhere with the new email.
- Q: Should the email-change initiation endpoint enforce application-level rate limiting? → A: No application-level rate limit — rate limiting is delegated entirely to edge infrastructure (Oathkeeper / Kratos / upstream throttling). The feature does not introduce its own per-user or per-(user, email) counters.
- Q: How does a platform administrator learn the outcome of an admin-initiated email change after the target user confirms (or fails to confirm)? → A: The admin polls a GraphQL status query under the `platformAdmin` surface that returns the current state of pending and recently-terminated changes (states: `initiated`, `confirmed`, `committed`, `rolled_back`, `expired`, `superseded`). No in-app, push, or email notification is introduced for the admin. The audit log remains the system of record.
- Q: When the identity-provider write fails during commit, should the platform retry before rolling back? → A: Bounded synchronous retry with backoff (2–3 attempts, total budget ~5–10 seconds) before rolling back. Beyond the budget, the failure is treated as terminal and the existing compensating-rollback path runs unchanged. No asynchronous / background reconciliation.
- Q: What locale should be used for the confirmation message and the security-signal notification? → A: Always English for both messages. No localization logic, locale lookup, or per-user locale resolution is introduced by this feature.
- Q: Can the subject user see (or cancel) a pending email change against their account before the new-mailbox confirmation completes? → A: View-only. The subject user can read their own pending change (including who initiated it: self vs platform admin) via a `me`-shaped GraphQL query. They cannot cancel it; not confirming is the only way to let it expire. No proactive notification of an admin-initiated change is sent ahead of the confirmation message at the new address.
- Q: How is the confirmation message delivered, and what does the confirm mutation accept? → A: Clickable link only. The confirmation message contains a URL carrying an opaque single-use token; the confirm mutation accepts a single token string. No code-based path is offered.
- Q: How long after a pending email change reaches a terminal state should it remain queryable via the `platformAdmin` status query and the subject user's `me`-shaped pending-change query? → A: 30 days. Pending-change records are retained for 30 days after entering a terminal state (`committed`, `rolled_back`, `expired`, `superseded`) and then removed; the raw audit log remains the system of record for longer-term investigation.
- Q: What information must the post-commit security-signal notification to the old email contain? → A: Statement of change, commit timestamp, initiator role (self vs platform admin), partially masked new address (e.g., `j***@e***.com`), and recovery instructions. The full new address, the confirmation token, and internal identifiers MUST NOT be disclosed.
- Q: What happens when the compensating rollback itself fails after its retry budget is exhausted? → A: The rollback observes the same bounded retry budget as the commit (2–3 attempts, ~5–10 s). On exhaustion the pending change transitions to a seventh terminal state `drift_detected`, an audit entry with outcome `drift_detected` is written (capturing observed per-side values and the failure reason), a distinct error is surfaced to the initiator identifying that drift remains and that manual reconciliation is required, and an operations-alert signal is emitted. No asynchronous reconciler is introduced; reconciliation is a manual op.
- Q: What happens if the outbound mail provider returns a hard failure during initiation, after validation has passed? → A: Atomic initiation. The just-persisted pending change is rolled back and the token discarded (treated as never issued); an audit entry with outcome `initiation_failed` is written describing the mail-send failure; the initiator receives a distinct mail-delivery error and may retry immediately as a fresh initiation.
- Q: Are users who authenticate exclusively via OIDC (no Kratos password credential) in scope for the email-change feature? → A: In scope, treated identically. All Kratos identities are eligible regardless of credential type (password, OIDC link, or both); the email-trait change is applied the same way; OIDC link bookkeeping is unchanged (consistent with Out of Scope).
- Q: By what mechanism is the FR-009a "operations-alert signal" emitted when a pending change transitions to `drift_detected`? → A: Existing observability channels only — a Winston error-level log entry and an Elastic APM error capture, both tagged with a known event marker (e.g., `email_change_drift_detected`). No new ops-alert infrastructure (RabbitMQ queue, webhook, paging integration) is introduced.
- Q: Does the confirmation-message link target a client-web URL or a server-side HTTP endpoint? → A: A client-web URL. The link is a client-web deep link carrying the opaque token; client-web reads the token and invokes the GraphQL confirm mutation. The server adds no new HTTP route; all confirmation logic remains behind the GraphQL surface.
- Q: Where do audit entries (FR-014) live? → A: In a dedicated `email_change_audit_entry` entity introduced by this feature (PostgreSQL-backed). It persists beyond the 30-day pending-change retention window (FR-020) and captures events with no corresponding pending-change record (validation rejects, FR-019a initiation rollback). The existing `activity-log` infrastructure MUST NOT be reused; the codebase has no general-purpose audit entity to extend. Winston log entries may be emitted in parallel for ops visibility but are not the system of record.
- Q: Is the confirmation token stored as plaintext or as a hash in the pending-change record? → A: Plaintext. The token is persisted directly on the pending-change record (no separate hash column); confirmation compares the presented token against the stored plaintext value. Safety relies on short TTL, single-use, supersession invalidation, and DB access controls. Audit entries MUST NOT contain the token value.
- Q: How long should `email_change_audit_entry` records be retained? → A: Indefinitely. This feature introduces no automatic purge for audit entries; GDPR right-to-erasure for a specific subject is handled by the existing user-deletion workflow, not by this feature's audit table.
- Q: Are `email_change_audit_entry` records queryable via GraphQL, and at what surface? → A: Yes. A minimal `platformAdmin` paginated GraphQL query keyed by subject user returns that subject's audit entries, ordered by timestamp descending. No additional filters (initiator, outcome, date range) in this iteration. No `me`-shaped audit query is introduced for the calling user.
- Q: By what mechanism does an admin reconcile a `drift_detected` pending change? → A: A dedicated `platformAdmin` GraphQL mutation introduced by this feature is the only supported recovery path. It accepts the subject user and an admin-chosen canonical email (one of the two observed values), force-aligns both sides within the FR-009 retry budget, and writes a `drift_resolved` audit entry. Direct PostgreSQL edits and direct Kratos admin-console edits MUST NOT be used as the operational recovery path. The `drift_detected` pending-change record itself remains in its terminal state as forensic evidence.
- Q: When the subject user reads their own active pending change via FR-022 and the initiator is a platform admin, does the response identify which admin? → A: Yes — a minimal admin profile summary (admin user id + display name) is included alongside the role so the subject user can identify the initiating admin. Admin email and other PII MUST NOT be included. The post-commit security-signal notification (FR-016) remains role-only and is not changed.
- Q: What is the confirmation token's time-to-live (TTL)? → A: 1 hour from issue timestamp, applied uniformly to both self-initiated and admin-initiated changes (no tiered TTL by initiator role). The TTL is now a contracted value (per FR-007b), superseding the earlier framing that treated it as tunable / not a contract surface.
- Q: How should the platform handle a failure to send the post-commit security-signal notification (FR-016) to the old email address? → A: Bounded synchronous retry (2–3 attempts, ~5–10s budget) against the mail channel, then on exhaustion write an audit entry with a new outcome `security_signal_failed` (capturing the underlying failure reason in the non-leaky field) and continue — the commit stands. No background retry is introduced; out-of-band admin contact is the reconciliation path.
- Q: How should the platform handle a failure to invalidate existing identity-provider sessions on commit (FR-017)? → A: Bounded synchronous retry (2–3 attempts, ~5–10s budget) against the identity provider, then on exhaustion write an audit entry with a new outcome `session_invalidation_failed` and continue — the commit stands and is not rolled back. No background retry; out-of-band admin recovery via the existing identity-provider admin surface. The residual risk (previously-established sessions surviving until natural expiry or manual invalidation) is an audited degradation of SC-009 in this failure mode.
- Q: When a pending email change is in the `drift_detected` terminal state, does the `me`-shaped query (FR-022) return it to the subject user? → A: Yes — FR-022 returns `drift_detected` records (within the 30-day FR-020 window) with an explicit indicator that admin reconciliation is required. Per-side observed email values, technical failure reasons, and other internal diagnostics MUST NOT be exposed by FR-022; those remain `platformAdmin`-only via the audit query (FR-014b). Other terminal states (`committed`, `rolled_back`, `expired`, `superseded`) are still not returned by FR-022.
- Q: What is the minimum entropy requirement for the confirmation token (FR-007a)? → A: ≥128 bits of cryptographically-random entropy, URL-safe serialization (e.g., base64url or hex). The token MUST NOT encode or derive from the subject user id, the proposed new email, the issue timestamp, or any other identifier — it is an opaque random value, and binding is via the pending-change record. The 128-bit floor is a contract surface; raising it is permissible without a spec change, lowering it is a spec-level decision.
- Q: What shape do the `initiator` and `subject` fields take in the FR-014b paginated audit-entries query response? → A: Minimal user profile summary `{ id, displayName }` for both (matching the FR-022 pattern), plus a separate `initiatorRole` tag (`self` vs `platform admin`). Email addresses and other PII MUST NOT be returned. For entries whose initiator cannot be resolved to a profile summary (e.g., early validation-rejection entries), the planning artifact MUST define a sentinel representation; `initiatorRole` MUST always be present. The canonical audit-outcome enumeration is now contracted in FR-014b.
- Q: Should the platform re-validate the proposed new email's uniqueness at confirmation time, in addition to at initiation? → A: Yes — re-check uniqueness at confirm before any side-write (same scope as FR-004, same case-insensitivity rules from FR-006). On now-taken, reject with a `conflict` error per FR-015 and write a `conflict` audit entry per FR-014, with no side-write on either side. This narrows the window in which the FR-009 compensating-rollback path is exercised for a benign uniqueness race between initiation and confirmation. Contracted in FR-004a.
- Q: What content must the confirmation message to the new mailbox include beyond the clickable link (FR-003a)? → A: Initiator role tag (`self` vs `platform admin`), an explicit statement that the link is time-limited consistent with the 1-hour TTL (FR-007b), and a recovery / disclaimer line for unexpected changes. No initiating admin name or other admin PII (role tag only — intentionally narrower than the FR-022 admin-profile disclosure), no confirmation token outside the link itself, no internal identifiers. English-only per FR-016a. Contracted in FR-003b. The trust-signal pattern mirrors FR-016's content contract for the post-commit notification at the old address.
- Q: Where in the GraphQL schema is the confirm-email-change mutation placed, and what authentication does it require? → A: A top-level (root) mutation, callable without an Alkemio session. The token is the sole authority; the resolver MUST NOT require or rely on any caller session for authorization. This is distinct from FR-018's `me` / `platformAdmin` placements for the two initiation mutations and matches the edge-case contract that confirmation succeeds regardless of session state (absent, expired, or belonging to a different account). Contracted in FR-018a.

## Assumptions

- The Alkemio user is already linked to a unique identity-provider identity id stored on the user record (stated in the issue: "needs the user's Kratos identity id, which we already store on the Alkemio user").
- The platform already operates an outbound mail capability suitable for sending confirmation messages and security-signal notifications.
- "Platform administrator" for the purpose of this feature means an actor with platform-admin privilege as defined by the existing authorization model. Community / space administrators do NOT inherit this capability, consistent with the project's `platformAdmin` GraphQL placement convention.
- The confirmation token's lifetime is contracted at 1 hour from issue timestamp per FR-007b (uniform across self-initiated and admin-initiated changes). This supersedes earlier framing that treated the TTL as tunable / not a contract surface.
- Email comparison for equality and uniqueness is case-insensitive on the local part as well as the domain. (This is permissive relative to RFC 5321, but aligns with how mainstream mail providers behave and with the identity provider's own behavior on the `email` trait.)
- "Atomic in effect" does not require a distributed transaction. A compensating-action approach is acceptable as long as the externally observable outcome matches the all-or-nothing requirement and the audit trail captures any compensations performed.
- Notifications to the old email and the session-invalidation policy are resolved per the Clarifications section (commit-time notification; invalidate-all-sessions on commit).
- Rate limiting for the initiation endpoint is provided by the existing edge infrastructure (Oathkeeper / Kratos / upstream throttling); the feature itself does not introduce per-user or per-(user, proposed-email) application-level counters or cooldowns. Planning MUST NOT add a rate-limit data store or cooldown column on the pending-change record.

## Dependencies

- The identity provider's admin API supports replacing an identity's `email` trait and the verified-address bookkeeping for an existing identity (i.e., updating in place by identity id), as documented in the issue's "Kratos behavior" section.
- An outbound mail channel is available for confirmation messages and security-signal notifications.
- Existing authorization and audit infrastructure can be extended to cover the new operations.
