# Implementation Plan: K8s Deployment Pipeline with Dual Docker Images

**Branch**: `081-k8s-deploy-pipeline` | **Date**: 2026-03-19 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/081-k8s-deploy-pipeline/spec.md`

## Summary

Build a production-grade CI/CD pipeline that produces two distinct Docker images per build — a distroless runtime image and an Alpine-based migration image — and uses a proper k8s Job (not a CronJob) for database migrations that must complete before the service is deployed. All GitHub Actions are updated to their latest stable pinned versions and all runners are moved to self-hosted `m4` tags. Three in-scope environments (dev auto-deploy, test manual, sandbox manual) share the same pipeline structure. The DockerHub release workflow is extended to publish both images.

## Constitution Check

_GATE: Must pass before Phase 0 research. Re-check after Phase 1 design._

This feature modifies only CI/CD infrastructure (Dockerfiles, GitHub Actions YAML, k8s manifests). It introduces zero NestJS source code changes. Principles 1–8 are not applicable to this work. The two directly relevant principles are evaluated below.

### Principle 9 — Container & Deployment Determinism ✅ SATISFIED

> Container images MUST be reproducible: explicit base image tags, no implicit latest pulls at runtime. Build outputs are immutable.

- Runtime image: `gcr.io/distroless/nodejs22-debian13:nonroot` — explicit tag, no `:latest`
- Builder stage: `node:22.21.1-alpine` — exact version, matching Volta config
- Migration image: `node:22.21.1-alpine` — same explicit tag
- The `:latest` tag is pushed to the dev environment only as a convenience alias; k8s deployments always reference the SHA tag
- `NODE_OPTIONS`, `NODE_ENV`, and port env vars are set via Dockerfile ENV (not process.env reads outside bootstrap)

### Principle 10 — Simplicity & Incremental Hardening ✅ SATISFIED

> Prefer the simplest viable implementation. Architectural escalation requires written rationale.

- `envsubst` for image substitution — a POSIX standard tool, no extra dependencies
- Raw `kubectl` commands for Job lifecycle — transparent and debuggable
- `azure/k8s-deploy` retained for the deploy step where image substitution in a Deployment is its primary use case
- No new abstractions, no custom GitHub Actions, no Helm templating introduced

### Constitution Check — Post-Design Re-evaluation ✅ NO VIOLATIONS

No unjustified deviations. The migration image intentionally uses Alpine (not distroless) because the migration entrypoint requires pnpm, which is unavailable in distroless — this is explicitly documented in `research.md` section 3.

## Complexity Tracking

> No constitution violations — this section is informational only.

| Design Choice | Why | Alternative Rejected |
|---|---|---|
| Migration image uses Alpine, not distroless | `pnpm run migration:run` requires a shell + pnpm on PATH | Standalone `node dist/migrate.js` entrypoint would require a new file and higher implementation risk |
| `envsubst` for Job image substitution | No extra tooling; POSIX standard | `kubectl set image` doesn't apply to pending Jobs; Helm adds scope; `azure/k8s-deploy` Job support is untested |
| Separate `Dockerfile.migration` file | Clear separation of concerns; no conditional build-arg logic | Multi-target single Dockerfile (using `--target`) would work but is less obvious for two semantically distinct images |
| `azure/k8s-deploy` retained for deploy step | It handles image substitution + rollout in Deployment cleanly | Raw `kubectl set image` + `kubectl rollout status` would also work but adds more boilerplate |

## Implementation Notes

### Dockerfile Changes

**Before** (single-stage Alpine): includes `src/`, `tsconfig*.json`, dev tooling, pnpm, all devDependencies — ~500–700 MB estimated.

**After** (multi-stage distroless runtime):
1. Builder stage (`node:22.21.1-alpine`): full build + `pnpm prune --prod` to strip devDeps
2. Runtime stage (`gcr.io/distroless/nodejs22-debian13:nonroot`): copies only `dist/`, `node_modules/` (prod), `package.json`, `alkemio.yml`
3. Expected size reduction: ≥40% (SC-003), primarily from eliminating src/, tsconfig, dev packages, and the Alpine OS layer

**Critical distroless CMD detail**: The `distroless/nodejs22` image already sets `ENTRYPOINT ["node"]`. Therefore:
```dockerfile
# CORRECT
CMD ["dist/main.js"]           # → runs: node dist/main.js

# WRONG — would try to run: node node dist/main.js
CMD ["node", "dist/main.js"]
```

### k8s Migration Job Pattern

The new pipeline's migrate step replaces the old CronJob-clone pattern:

```bash
# Old approach (3 steps, semantically wrong)
azure/k8s-deploy → update CronJob image
kubectl delete job server-migration-job
kubectl create job --from=cronjob/server-migration server-migration-job

# New approach (3 steps, semantically correct)
kubectl delete job server-migration-job --ignore-not-found=true
MIGRATION_IMAGE=<registry>/alkemio-server-migration:<sha> envsubst < manifests/27-server-migration-job.yaml | kubectl apply -f -
kubectl wait --for=condition=complete job/server-migration-job --timeout=300s
```

The `backoffLimit: 0` + `restartPolicy: Never` combination means a failing pod immediately marks the Job as `Failed`. The `kubectl wait --for=condition=complete` exits 1 on timeout (which occurs as soon as the Job reaches `Failed` state and the condition `complete` will never be satisfied). This causes the migrate step to fail, and `needs: migrate` causes the deploy job to be skipped.

## Risk Assessment

| Risk | Likelihood | Mitigation |
|---|---|---|
| `pnpm prune --prod` removes a dep that is actually needed at runtime | Low | Verify by running the app container locally; TypeORM and all service integrations are prod deps |
| `kubectl wait --for=condition=complete` hangs 300s on a fast-failing job | Medium | `backoffLimit: 0` makes the Job fail immediately; the wait exits at timeout. Pipeline is blocked for ≤300s, not permanently. |
| `envsubst` substitutes unintended `$VARIABLE` tokens in the Job YAML | Low | The Job manifest uses only `${MIGRATION_IMAGE}` as a placeholder; other env vars are sourced from k8s ConfigMaps/Secrets, not shell vars |
| `gcr.io/distroless` image pull fails in air-gapped or restricted cluster | Low | Clusters are on Hetzner with public internet access; not a known restriction |
| Native Node.js modules (if any) compiled for Alpine glibc vs. distroless Debian | Low-Medium | Distroless uses Debian/glibc; Alpine uses musl. If the builder is Alpine and final is distroless-Debian, native modules compiled for musl may fail. Mitigation: use `node:22.21.1-bookworm-slim` as builder if native module issues appear. |

> **Note on native modules**: The existing Dockerfile uses Alpine. If any native module is compiled for musl libc in the builder stage and then copied to the distroless Debian (glibc) stage, it will fail at runtime. This is the primary technical risk. If observed, the builder stage should be changed to `node:22.21.1-bookworm-slim` to match the Debian base of the distroless image. This should be validated during implementation.
